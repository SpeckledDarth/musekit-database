Prompt 30: Create site_pages Table in Supabase
Context
Production Target
Supabase PostgreSQL database
This module lives at packages/database/ in a Turborepo monorepo
The site_pages table was referenced in prompt 08 as "TO BE CREATED" but was never actually created
Multiple features depend on this table: the CMS HomePageLoader, dynamic page routing (/[slug]), the admin Pages setup page, and the SiteNav component
Problem: Missing Table
The site_pages table does not exist in the Supabase database. Without it:

The HomePageLoader component fails to fetch homepage section configuration and falls back to hardcoded defaults
Dynamic page routing (apps/web/src/app/[slug]/page.tsx) cannot resolve custom pages
The admin setup Pages page (/admin/setup/pages) cannot create, edit, or list site pages
The SiteNav component cannot query pages with show_in_nav = true
Impact
The entire CMS page management system is non-functional
Homepage sections cannot be configured from admin (toggle on/off, reorder)
No custom pages can be created (about, contact, etc.)
Fix — Create Table, RLS Policies, and Seed Data
Step 1: Create the site_pages table
Run this SQL in the Supabase SQL Editor (or add as a migration file):

CREATE TABLE IF NOT EXISTS site_pages (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug text UNIQUE NOT NULL,
  title text NOT NULL DEFAULT '',
  content text DEFAULT '',
  sections jsonb DEFAULT '[]'::jsonb,
  meta_title text DEFAULT '',
  meta_description text DEFAULT '',
  status text NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'published')),
  show_in_nav boolean DEFAULT false,
  sort_order integer DEFAULT 0,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);
CREATE INDEX idx_site_pages_slug ON site_pages(slug);
CREATE INDEX idx_site_pages_status ON site_pages(status);
Step 2: Add RLS Policies
ALTER TABLE site_pages ENABLE ROW LEVEL SECURITY;
-- Public read access for published pages
CREATE POLICY "Anyone can read published site pages"
  ON site_pages FOR SELECT
  USING (status = 'published');
-- Admin full access (using service role key bypasses RLS, but this policy
-- allows admin users authenticated via anon key to also manage pages)
CREATE POLICY "Admins can manage site pages"
  ON site_pages FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM profiles
      WHERE profiles.id = auth.uid()
      AND profiles.role IN ('admin', 'super_admin')
    )
  );

Critical Rules
Run the CREATE TABLE SQL in Supabase first — the migration file is for version control, the live database needs the table now
Do NOT modify any existing tables — only create the new site_pages table
The slug column must be UNIQUE — the CMS queries by slug for page routing
The sections column must be jsonb — it stores the homepage section configuration as a JSON array
Seed the home entry with status = 'published' — this is required for HomePageLoader to find it
The sections JSON format must match what HomePageLoader expects: [{type, enabled, sortOrder, config}]
